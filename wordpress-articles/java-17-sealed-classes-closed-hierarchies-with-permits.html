In object-oriented Java, inheritance has always been open by default: any class can extend another (unless you mark it final). That flexibility is powerful, but it creates a long-term problem for library authors and domain modelers: you never know the full set of subtypes that might exist at runtime. Framework code that switches on types, serializes polymorphic graphs, or validates business rules cannot rely on the compiler to catch a missing case when someone adds class Surprise extends Payment in another module six months later.

Sealed classes, delivered in Java 17 (JEP 409), flip part of that model: you can declare a closed hierarchy—a fixed list of permitted subtypes—using the sealed modifier and a permits clause. The compiler then enforces membership and modifier rules at compile time. Together with pattern matching (standardized in later Java releases), sealed types are one of the most practical language features for modeling algebraic data shapes in plain Java without extra frameworks.

This article walks through how permits works, what the three permitted-subclass modifiers mean, how nesting and escape hatches fit together, and how to try a runnable example you can clone from the codingstrain repository.

What changes with sealed types?

Consider a small geometry API. You want exactly three families of shapes under a common Shape abstraction: circles, rectangles (with known subtypes), and a “fancy” rectangle branch that third parties may extend. Without sealed types, you document the intended hierarchy in Javadoc and hope teams respect it. With sealed types, you encode the contract in the language:

public abstract sealed class Shape
    permits Circle, Rectangle, FancyRectangle {

    public abstract double area();
    public abstract String label();
}

Only Circle, Rectangle, and FancyRectangle may extend Shape. Any other declaration—class Triangle extends Shape—is a compile-time error, even if the class is in another package and even if it is never instantiated.

That is the core idea: the permitted subclasses are part of the API contract, checked by javac, not merely a team convention.

The permits clause

The permits list names every direct subclass of the sealed class or interface. It is not transitive: if Rectangle is permitted by Shape, the subtypes of Rectangle are governed by Rectangle’s own permits clause (when Rectangle is itself sealed).

You can omit permits when all permitted subclasses are declared in the same source file as the sealed type; the compiler infers them. In real projects, explicit permits is clearer and survives refactoring across files.

Sealed types work on classes and interfaces. Sealed interfaces are especially useful for state machines, command/event models, and result types (Success | Failure) where the set of variants must stay small and known.

Three modifiers for permitted subtypes

Every class listed in a permits clause must be declared with exactly one of three modifiers. This is a second, easy-to-miss compiler rule:

ModifierMeaningTypical use
finalNo further subclasses.Leaf variants (Circle, Square).
sealedAnother closed level; must declare its own permits.Intermediate grouping (Rectangle → Square, Oblong).
non-sealedStops sealing here; ordinary open inheritance resumes below.Extension point for plugins or user code.

If you list a class in permits but forget all three modifiers, compilation fails with a message that the permitted subtype must be final, sealed, or non-sealed. That prevents “leaky” hierarchies where a permitted class could still be extended arbitrarily.

Building a nested hierarchy: Shape example

The following structure matches the diagram and sample project in x-posts-examples/java-17-sealed-classes:

Shape (abstract sealed)
├── Circle (final)
├── Rectangle (sealed, permits Square, Oblong)
│   ├── Square (final)
│   └── Oblong (final)
└── FancyRectangle (non-sealed)
    └── RoundedRectangle (ordinary class — not in Shape.permits)

Circle is a leaf: final class Circle extends Shape closes the branch.

Rectangle is a nested sealed type:

public sealed class Rectangle extends Shape permits Square, Oblong {
    // width, height, area() ...
}

Only Square and Oblong may extend Rectangle. Both are final, so the rectangle family is fully closed.

FancyRectangle uses the escape hatch:

public non-sealed class FancyRectangle extends Shape {
    // ...
}

Because FancyRectangle is non-sealed, any class may extend it—RoundedRectangle does not appear in Shape’s permits list and that is legal. The global hierarchy is closed at the Shape level (only three direct children), but the subtree under FancyRectangle is open again for frameworks or customer extensions.

Choosing non-sealed is deliberate: you keep a stable, enumerable set of top-level variants while allowing unknown subtypes below one branch—similar to how Exception is open but many domain enums are not.

What the compiler verifies

At compile time, the compiler checks at least the following:

  • Membership: every direct subclass of a sealed type appears in permits (or is inferred in the same file).
  • Modifier on permitted types: each permitted class is final, sealed, or non-sealed.
  • Sealed subclasses with children: a sealed class that has subclasses must declare permits (or same-file inference).
  • No stray extenders: a class not in the list cannot extend the sealed type.

These checks turn design rules into errors you see in the IDE, not surprises in production when a new JAR appears on the classpath.

Why sealed hierarchies matter for application design

Domain modeling. Payment types, order statuses, notification channels, and AST node kinds are classic “sum type” shapes: a value is exactly one of several alternatives. Sealed classes express that intent in Java without bolting on extra libraries.

Exhaustive control flow. When the compiler knows all subtypes of a sealed type, it can verify that a switch or pattern match covers every case. In Java 21 and later, pattern matching for switch is standard; on Java 17 you still benefit from clearer APIs and from instanceof chains where the set of types is documented and enforced. Fewer default branches that silently swallow new variants means fewer production bugs when someone adds class CryptoPayment extends Payment without updating the fee calculator.

API stability. Library maintainers can evolve a sealed hierarchy additively (by adding a new permitted subtype in a minor release) while making it obvious to consumers that the set is curated. Contrast that with open inheritance, where any public non-final class is implicitly extensible forever.

Security and auditing. Closed hierarchies simplify reasoning about deserialization, reflection, and policy checks: “only these classes may represent a Credential.”

Common mistakes and compile errors

Undeclared subclass.

class Triangle extends Shape { } // error: cannot extend sealed Shape

Permitted but missing modifier.

// abstract sealed class Shape permits Bad { }
class Bad extends Shape { } // error: must be final, sealed, or non-sealed

Sealed intermediate without permits. If Rectangle has subclasses but no permits clause (and they are not in the same file), compilation fails.

Assuming permits is transitive. Listing Rectangle under Shape does not automatically permit Square; Rectangle must seal its own children explicitly.

Sealed classes vs other approaches

Enums work when variants are singletons without per-case fields or when behavior fits in enum constants. Sealed classes fit when each variant carries different state (Circle has radius, Oblong has width and height).

Final-only base classes prevent extension entirely. Sealed types are the middle ground: curated extension, not zero extension.

Open inheritance + documentation is the status quo sealed types improve on; the compiler simply did not participate before Java 17.

Practical guidelines

  • Start with the smallest closed set that models your domain; add variants deliberately.
  • Use final for true leaves, sealed for grouping, and non-sealed only where you genuinely need an open plugin line.
  • Keep permits explicit in multi-file modules for readability and code review.
  • Pair sealed roots with exhaustive switch (Java 21+) or disciplined instanceof handling in libraries still on 17.
  • Document migration: adding a permitted subtype is a source-compatible change for callers but may require updating switches—treat it like an API change.

Conclusion

Java 17 sealed classes turn “these are all the subtypes” from a comment into a compile-time guarantee. The permits clause defines who may join the hierarchy; final, sealed, and non-sealed describe what each member may do next. Nested sealing models structured domains; non-sealed preserves escape hatches where openness is still required.

If you model payments, events, AST nodes, or API results as class hierarchies, sealing the root (and sealing intermediate groupings where it helps) is one of the highest-leverage Java 17 upgrades you can adopt without touching frameworks or build tools. Clone the sample project, break the rules on purpose to see the compiler errors, and export the hierarchy diagram for your next post—you will remember the three modifiers faster than any cheat sheet.
